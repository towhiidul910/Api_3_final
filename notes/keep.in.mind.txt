Zod first validates:
v = whatever the input was
"20" ✅ is a string

undefined ✅ allowed because of .optional()

"abc" ✅ is a string (still passes the z.string() check at this point)

Then the transform runs:

If v = "20" → parseInt("20") = 20

If v = undefined → returns 10

If v = "abc" → parseInt("abc") = NaN 🤢 (Zod won’t complain here unless you add a .pipe(z.number()))

The transformed value is now what your schema outputs.

8. 🦕orderBy
What orderBy is in Prisma

orderBy controls the sorting order of your query results. 
It’s basically the Prisma version of SQL’s ORDER BY. You pick a field (like createdAt, name, version, etc.) and say whether results should come in ascending (asc) or descending (desc) order.

for number & dates:
desc = descending (biggest/newest first)
asc = ascending (smallest/oldest first)

for string:
asc → A → Z (alphabetical order)
desc → Z → A (reverse alphabetical)

where: {
  OR: [
    { language: "English" },
    { version: { gte: 2 } }
  ]
}

This means:

User can be English, OR
User can have version ≥ 2

So:

English, version 1 → ✅ accepted
Spanish, version 3 → ✅ accepted
Spanish, version 1 → ❌ rejected

13. 🦕Before: N:M steps to take: 
Connecting existing records

            connect vs create — e.g., link an existing post to a user without creating a duplicate.

Updating related records

            update + nested update for posts/comments.

            Example: update a post’s title while fetching the user.

Deleting with cascade

            When you delete a User, what happens to Posts?

            Optional: onDelete: Cascade in schema.

1:1 relations

            Address & Geo, Company & User — practice creating/updating both sides.

            Make sure you’re comfortable with create, connect, update in 1:1.

Sequence awareness

            You already learned the hard way: nested writes + manually inserted rows = P2002.

            Understand why sequences matter before jumping to N:M.
