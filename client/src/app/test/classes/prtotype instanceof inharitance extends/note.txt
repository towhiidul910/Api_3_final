## 1️⃣ How `ZodError` looks in practice

Imagine you have this schema:

```ts
import { z, ZodError } from "zod";

const userSchema = z.object({
  username: z.string(),
  age: z.number().min(18),
});

try {
  userSchema.parse({
    username: 123,   // ❌ wrong type
    age: 15,         // ❌ too small
  });

  if (err instanceof ZodError) {
    console.log(JSON.stringify(err, null, 2));
```

## 2️⃣ Classes under the hood

Simplified version of ZodError class (not exact code, but conceptually):

```ts
class ZodError extends Error {
  issues: ZodIssue[];

  constructor(issues: ZodIssue[]) {
    super(`${issues.length} validation error(s)`);
    this.name = "ZodError";
    this.issues = issues;
    Object.setPrototypeOf(this, ZodError.prototype); // ensures instanceof works
  }

  format() {
    const formatted: Record<string, string[]> = {};
    for (const issue of this.issues) {
      const path = issue.path.join(".");
      if (!formatted[path]) formatted[path] = [];
      formatted[path].push(issue.message);
    }
    return formatted;
  }
}

type ZodIssue = {
  code: string;
  path: (string | number)[];
  message: string;
  [key: string]: any;
};
```

**Takeaways:**

* `ZodError` extends **native `Error`** → inherits `stack` and `message`.
* `issues` array contains **all the validation errors**.
* `format()` → converts errors to an **API-friendly object**.
* `Object.setPrototypeOf` ensures `instanceof ZodError` works even though it extends a built-in class.

---

### TL;DR

* `ZodError` is a class that extends `Error`.
* It has **issues array**, `name`, `message`, and `stack`.
* `.format()` or `.flatten()` → makes it easier to return structured errors in API.
* `instanceof ZodError` works because it’s a proper class with prototype chain intact.

🦕instanceof

Ah, now we’re getting into the **exact mechanics**. Let’s represent `instanceof` **visually in text** so you can see why it works and what it actually checks.

---

### 1️⃣ Prototype chain (text diagram)

When you do:

```ts
class Animal {}
class Dog extends Animal {}

const myDog = new Dog();
```

The **prototype chain** looks like this:

```
myDog
  │
  └─> Dog.prototype
        │
        └─> Animal.prototype
              │
              └─> Object.prototype
                    │
                    └─> null
```

`instanceof` walks **up this chain** to see if the constructor’s `.prototype` exists anywhere.

---

### 2️⃣ How `instanceof` works (step-by-step)

```ts
myDog instanceof Dog
```

* Checks: `myDog.__proto__ === Dog.prototype` → true ✅
* Stops because match found.

```ts
myDog instanceof Animal
```

* Checks: `myDog.__proto__ === Animal.prototype` → false
* Goes up: `myDog.__proto__.__proto__ === Animal.prototype` → true ✅
* Returns `true`

```ts
myDog instanceof Object
```

* Walks up → eventually finds `Object.prototype` → true

---

### 3️⃣ ZodError example

```
err (instance of ZodError)
  │
  └─> ZodError.prototype
        │
        └─> Error.prototype
              │
              └─> Object.prototype
                    │
                    └─> null
```

When you do:

```ts
if (err instanceof ZodError)
```

`instanceof` checks:

1. `err.__proto__ === ZodError.prototype?` → yes → true ✅
2. If no, move up: `err.__proto__.__proto__ === ZodError.prototype?` → repeat until `null`.

---

### 4️⃣ Important notes

* **`instanceof` does NOT check for properties like `issues`.**

  * `err.issues` existing is **not why** `instanceof ZodError` works.
  * It **checks the prototype chain** only.
* That’s why random objects with an `issues` property **won’t pass `instanceof ZodError`**.

---

### TL;DR (text visual summary)

```
instanceof checks the prototype chain, NOT the properties:

err instanceof ZodError
  └─> Does err.__proto__ === ZodError.prototype?
       └─> If no, go up chain: err.__proto__.__proto__ ...
            └─> Stops at null
  → returns true if ZodError.prototype found anywhere
```

* ✅ Checks **structure of inheritance**, not properties like `issues`.
* ✅ That’s why it works reliably with ZodError objects.

🦕prototype
Alright bro, let’s strip it down to the basics. **Prototype** is one of those JS concepts that *makes inheritance and `instanceof` work*.

---

### 1️⃣ What a prototype is (simple version)

In JavaScript, **every object has a hidden link to another object**, called its **prototype**.

* The prototype is like a **backup object**: if a property or method doesn’t exist on your object, JS looks in its prototype.
* This is how inheritance works in JS: **objects “borrow” properties and methods from their prototype**.

---

### 2️⃣ Example

```ts
class Animal {
  speak() {
    console.log("Animal speaks");
  }
}

const dog = new Animal();
dog.speak(); // Animal speaks
```

Behind the scenes:

```
dog
  │
  └─> Animal.prototype
        │
        └─> Object.prototype
              │
              └─> null
```

* `dog` itself doesn’t have `speak`. JS looks up the chain → finds it in `Animal.prototype`.
* That’s why `dog.speak()` works.

---

### 3️⃣ Prototype chain

Think of it as a **linked list**:

```
object → prototype → prototype → ... → null
```

* JS walks this chain **whenever you access a property or method**.
* The end of the chain is `Object.prototype`, which has common methods like `toString()`.

---

### 4️⃣ Prototype vs class

* Classes in JS are just **syntactic sugar** over prototypes.
* `class Animal {}` creates a **function** called `Animal`, and all instances automatically get `Animal.prototype` as their prototype.

```ts
console.log(dog instanceof Animal); // true
// instanceof checks: dog.__proto__ === Animal.prototype ? → yes
```

---

### 5️⃣ TL;DR

* **Prototype = “backup object”** for property/method lookup.
* Objects inherit properties/methods from their prototype chain.
* `instanceof` checks if a constructor’s prototype exists **anywhere in that chain**.
* Classes in JS are just **nice wrappers around prototypes**.
